soar:
  situation: >-
    A client running a high-volume eBay store needed every product image
    standardized — consistent square crop, padding color, logo placement, and
    border style — across the entire active inventory. Manual editing at that
    listing count was a non-starter.
  obstacle: >-
    The eBay Trading API enforces per-second AND per-day rate limits that would
    block a naïve batch uploader; the owner needed to preview padding/logo/border
    choices before committing; and any partial failure mid-batch would leave the
    store half-styled and hard to recover.
  action: >-
    I built a Python pipeline against the Trading API plus a companion Aurora
    processor web app for previewing style choices. I added intelligent caching
    so unchanged listings get skipped, exponential-backoff rate limiting to
    respect API quotas, an idempotent per-listing log so partial failures are
    recoverable, and a scheduled service to process new listings overnight.
  result: >-
    Visual consistency across the catalog with a 100% production success rate
    after the redesign, robust error handling, and a preview workflow that lets
    the owner iterate on style in seconds. The service now runs unattended.
failureStory:
  context: >-
    On the day the first version of the pipeline was finished, I tried to push
    the first big batch of standardized images through the eBay Trading API
    against the client's live store — same day, no dry-run, no staging.
  whatHappened: >-
    I hadn't fully modeled the per-second AND per-day rate limits, so the batch
    tripped a throttle midway through. Some listings updated, some silently
    didn't, and I had to scramble to figure out what was actually applied
    versus rejected — exactly the wrong way to discover a rate limit on a
    client's live store.
  learning: >-
    I now treat external APIs as first-class infrastructure. I build a dry-run
    mode and a rate-limit-aware queue BEFORE the code ever touches production,
    and I keep an idempotent log of every attempted change so partial failures
    are recoverable rather than catastrophic. That mindset is what made the
    eventual 100% success rate possible — and it's the lesson I now apply to
    every API integration I build.
